# 异构数据库数据传输 / 数据同步 / 结构同步方案

> 目标读者：后续实施异构（跨驱动，如 MySQL↔PostgreSQL）能力的开发者。
> 本文档描述现状边界、目标架构与分期路线图。接口契约背景见 `ARCHITECTURE.md` §3、§6.6、§6.7。
> 状态：P0 已完成（2026-07）；P1 起为待实施方案。

---

## 1. 背景与设计立场

项目已有两个 SQL 驱动（`plugins/mysqldrv`、`plugins/postgresdrv`），通用层通过 `internal/dbdriver` 抽象隔离方言。三个功能的异构就绪度不同：

| 功能 | 入口 | 异构现状 |
|---|---|---|
| 数据传输 | `TransferService.StartTransfer` | 行数据可用（多值参数化 INSERT）；建表跨驱动会重渲染但**类型未映射** |
| 数据同步 | `SyncService` + `core/datasync` | 架构可用（目标端 Editor 参数化）；**值比较归一化不足** |
| 结构同步 | `SyncService.CompareSchemas` + `core/schemadiff` | 同构完整；跨驱动 CREATE 有分叉但**类型未映射**，ALTER 级比对会产生假差异 |

**核心设计立场（贯穿全文）**：

1. **中间表示走加细后的 `LogicalType`，映射逻辑放各驱动**。每个驱动声明"我如何渲染每个逻辑类型"（O(N)），而不是维护两两驱动的映射表（O(N²)）。
2. **有损转换必须显式**。产出 warning 随 diff/传输预览给前端，绝不静默降级。
3. **值永远走绑定参数跨驱动**，字面量渲染只允许出现在 SQL dump 文件（`.sql` 文件天然无参数）。
4. **通用层零方言知识**。所有跨方言差异经 `Dialect`/`ScriptRules`/可选扩展接口自描述，`leakguard_test.go` 继续兜底。

## 2. 现状盘点（P0 修复后的基线）

### 2.1 已就位的构件

- **`createTableDDL` 同构/异构分叉**（`internal/services/sync_service.go:343`）：同构走源库 `GetCreateTable` 原生 DDL（保真度最高）；跨驱动读回 `TableSchema` 用目标 `Dialect.GenerateCreateTable` 重渲染。结构同步与数据传输建表共用。
- **`Dialect.Placeholder(i int)`**（`internal/dbdriver/driver.go`）：MySQL `?`、PG `$i`，通用层据此拼多值参数化 SQL。
- **多值参数化批量 INSERT**（`internal/services/transfer_service.go:697` `transferBatch` + `:717` `buildBatchInsert`）：按 `maxInsertParams = 60000` 分块（MySQL prepared statement 上限 65535 个占位符）；`bindArg`（`:753`）把 scanner 的 `BytesValue`/`BigIntString` 包装类型解包成 `[]byte`/数字串再绑定。
- **SQL dump 字面量渲染**（`transfer_service.go:436` `sqlLiteral`）：按 `ScriptRules` 选转义族——`BackslashEscapes`（MySQL 系）反斜杠加倍 + `X'hex'` 字节字面量；否则 standard-conforming（PG 系）+ `'\xhex'` bytea 字面量；bool 统一 `TRUE/FALSE`。
- **`DatabaseRouter` 多库路由**（`internal/dbdriver/driver.go:62`）：PG 按库懒建独立 pool，`RouteQuerier/RouteBegin` 已贯穿传输/同步/编辑/导入导出。
- **数据同步归并引擎**（`internal/core/datasync/datasync.go:98` `Merge`）：双端 `ORDER BY pk` 流式归并，写端全走目标 `Editor` 参数化语句 + 独立连接批量事务。

### 2.2 剩余缺口（即后文 P1–P3 的工作）

| 缺口 | 位置 | 影响 |
|---|---|---|
| 跨方言类型映射缺失 | 两驱动 `ddl.go` 均原样输出 `ColumnMeta.NativeType` | MySQL `int(11) unsigned`/`longtext`/`enum(...)` 渲染进 PG DDL 非法，跨驱动建表失败 |
| ALTER 级比对单端归一化 | `sync_service.go` `CompareSchemas` 用目标方言 `NormalizeType` 比两侧类型串 | 跨驱动几乎每列报假差异；生成的 ALTER 带源生类型，非法 |
| 值比较归一化不足 | `datasync.go:253` `normalize` | MySQL `tinyint(1)`→`int64(1)` vs PG `bool true` 恒判 update；时间/decimal 文本格式差异产生假差异 |
| 字符串主键排序错位 | `Merge` 依赖两端 `ORDER BY` 全序一致 | 跨库 collation 排序规则不同 → 归并错位（同库大小写不敏感 collation 已知，跨库更严重） |
| 批量写入性能 | `transferBatch` 逐语句 Exec | PG 可用 `CopyFrom`、MySQL 可用更大包，多值 INSERT 只是基线 |
| 视图/例程跨库 | `compareViews` 比对 SQL 原文 | 跨库必然全部"不同"，且无法转译 |

## 3. P1：跨方言类型映射管道（核心构件）

### 3.1 加细 `LogicalType` / 扩展 `ColumnMeta`

现有 `LogicalType`（`internal/dbdriver/types.go:82`）对 DDL 重建是有损的：`TypeInt` 不区分宽度、丢 unsigned、丢 enum 值。两条路线取其一：

**推荐：保持 `LogicalType` 枚举不动，映射函数直接吃 `ColumnMeta`**。`ColumnMeta` 已携带 `NativeType/Length/Precision/Scale/IsAutoIncrement`，缺的信息补进去：

```go
// ColumnMeta 增补（internal/dbdriver/types.go）
type ColumnMeta struct {
    // …现有字段…
    Unsigned   bool     `json:"unsigned,omitempty"`   // MySQL 数值修饰
    EnumValues []string `json:"enumValues,omitempty"` // ENUM/SET 的值清单
    IntWidth   int      `json:"intWidth,omitempty"`   // 8/16/24/32/64（bit 数，源驱动折算）
}
```

各驱动的 `Metadata.ListColumns/ListAllColumns` 负责填充（MySQL 从 `COLUMN_TYPE` 解析，PG 天然无 unsigned/宽度歧义）。**这是 Metadata 层的自描述扩展，不破坏现有调用方**。

### 3.2 `Dialect` 新方法：`RenderPortableType`

```go
// Dialect 接口增补（internal/dbdriver/driver.go）
//
// RenderPortableType 把一个（可能来自其他驱动的）列描述渲染成本方言的
// NativeType。losses 描述有损转换（如 "unsigned BIGINT → NUMERIC(20)"），
// 为空表示无损。仅在跨驱动路径调用；同构路径保持原生类型原样。
RenderPortableType(col ColumnMeta) (nativeType string, losses []string)
```

实现要点（每驱动 ~100 行 + 表驱动单测）：

- 入口先 `MapType(col.NativeType)` 得到 LogicalType，再按 LogicalType + 宽度/精度/unsigned/enum 渲染本方言类型；`TypeUnknown` 落到本方言的 `TEXT` 等价物并报 loss。
- **PG 侧关键映射**：`TINYINT(1)`→`BOOLEAN`（报 loss，见 §5 值语义联动）、`TINYINT/SMALLINT`→`SMALLINT`、`MEDIUMINT/INT`→`INTEGER`、`BIGINT`→`BIGINT`、`INT UNSIGNED`→`BIGINT`、`BIGINT UNSIGNED`→`NUMERIC(20)`（loss）、`DOUBLE`→`DOUBLE PRECISION`、`DATETIME(n)`→`TIMESTAMP(n)`、`TINYTEXT..LONGTEXT`→`TEXT`、`TINYBLOB..LONGBLOB`→`BYTEA`、`ENUM(...)`→`VARCHAR(最长值) + CHECK`（loss；不引入 `CREATE TYPE`，避免建表语句携带带状态的前置对象）、`SET(...)`→`VARCHAR + loss`、`JSON`→`JSONB`、`YEAR`→`SMALLINT`（loss）。
- **MySQL 侧关键映射**：`BOOLEAN`→`TINYINT(1)`、`SERIAL/IDENTITY`→`INT/BIGINT AUTO_INCREMENT`、`TEXT`→`LONGTEXT`、`BYTEA`→`LONGBLOB`、`NUMERIC`（无参）→`DECIMAL(65,30)`（loss）、`TIMESTAMPTZ`→`TIMESTAMP`（loss：丢时区语义）、`UUID`→`CHAR(36)`（loss）、数组/range/几何→`TEXT` + loss。
- 自增语义走 `IsAutoIncrement` 布尔，不看类型名（PG 渲染 `GENERATED BY DEFAULT AS IDENTITY`，MySQL 渲染 `AUTO_INCREMENT`——两驱动 `GenerateCreateTable` 已具备）。
- 默认值表达式跨方言（`CURRENT_TIMESTAMP` vs `now()`）第一期只透传可识别的常量与 `CURRENT_TIMESTAMP` 家族，其余丢弃并报 loss。

### 3.3 接入点

1. **`createTableDDL` 跨驱动分支**（`sync_service.go:343`）：`GenerateCreateTable` 前逐列过 `dia.RenderPortableType`，losses 汇总进 `SchemaObjectDiff` 新增的 `Warnings []string` 字段（传输侧进 `TableTransferResult.Warnings`），前端预览展示。索引/外键的列引用不动；MySQL 特有的索引前缀长度、`Options`（ENGINE/CHARSET）跨驱动直接剔除。
2. **数据传输建表**（`transfer_service.go` `StartTransfer`）：同上，自动获得该能力（已复用 `createTableDDL`）。
3. **契约测试**（`internal/dbdriver/contract`）：新增 `testPortableTypes` —— 用一张覆盖全部 LogicalType 的虚构 `TableSchema`（来自"另一个驱动"的 NativeType 串）过 `RenderPortableType` + `GenerateCreateTable`，在真实库执行建表并 `ListColumns` 回读，断言 `MapType(回读类型) == 期望 LogicalType`（round-trip）。

### 3.4 明确不做

- 不做 `CREATE TYPE`（PG enum）等有状态前置对象。
- 不做 collation/charset 映射（跨驱动剔除 + loss 提示）。
- 不做列注释以外的 `Options` 透传。

## 4. P1.5：结构同步 ALTER 级跨驱动比对

跨驱动 ALTER 是假差异重灾区，**第一期跨驱动结构同步只放行 CREATE**（目标缺表），存量表标记 `cross-driver-alter-unsupported` slug，UI 提示"跨驱动仅支持建表"。第二期再做：

1. `schemadiff.Options` 增加跨驱动比较模式：类型相等判定不再比 NativeType 串，改为**先把源列过目标 `RenderPortableType`，再用目标 `NormalizeType` 与目标列比较**——复用 P1 管道，无需第二套等价表。
2. 索引 `Type` 大小写/家族归一（`BTREE`≈`btree`）、FK action 语义归一已有基础（`schemadiff.Diff` 的 RESTRICT 折叠），补跨驱动缺省差异。
3. 默认值表达式比较第一期跳过（跨驱动模式下默认值差异只报 warning 不进 ChangeSet）。

## 5. P2：数据同步与传输的值语义

### 5.1 `datasync` 值归一化按 LogicalType 驱动

`normalize`（`datasync.go:253`）现覆盖整型宽度/float/`[]byte`↔string。扩展为**由列的 LogicalType 决定比较器**（`TableSync` 已持有双端 Dialect，`resolveTablePairs` 时把公共列的双端 LogicalType 算好传入）：

- `TypeBool`：`int64(0/1)`、`bool`、`"t"/"f"` 折叠为 bool（解决 MySQL `tinyint(1)` vs PG `boolean` 恒判 update）。
- `TypeDateTime/Timestamp`：双端文本解析为 `time.Time` 比较（MySQL `2006-01-02 15:04:05` vs PG 可能带时区偏移）；解析失败回退原串。
- `TypeDecimal`：去尾零后比较（`shopspring/decimal` 或手写规范化，避免新依赖优先手写）。
- `BigIntString`/`BytesValue` 包装类型先解包再比（复用 `services` 的 `bindArg` 逻辑，抽到 scanner 或 datasync 共用）。

### 5.2 前置校验扩展

`resolveTablePairs`（`internal/services/sync_data.go:102`）新增 skip slug：

- `type-incompatible`：公共列双端 LogicalType 不兼容（如源 `TypeJSON` vs 目标 `TypeInt`）→ 剔除该列或跳过表。
- `collation-order-risk`：跨驱动 + 字符串主键 → 第一期直接跳过（数值/二进制主键放行）。第二期可选方案：拉回后在 Go 侧用统一比较器重排缓冲窗口，代价是内存与实现复杂度，需单独评估。

### 5.3 传输性能（可选扩展接口）

```go
// 可选扩展（类型断言探测，同 BulkMetadata 模式）
type BulkInserter interface {
    BulkInsert(ctx context.Context, db, schema, table string,
        cols []string, rows [][]any) (int64, error)
}
```

PG 实现走 `pgx.CopyFrom`（注意 simple protocol 文本值需转成 pgx 可编码类型），MySQL 实现可加大 `maxInsertParams` 利用率或 `LOAD DATA LOCAL INFILE`。`transferBatch` 探测该接口，无则回退现有多值参数化 INSERT。**只有在基线性能实测不达标时才做**。

## 6. P3 / 非目标

- **视图、存储过程、触发器、事件的跨驱动转译**：本质是 SQL 方言翻译，明确不支持。跨驱动结构同步中这些对象标记 `cross-driver-unsupported`，UI 隐藏或置灰。
- **触发器/生成列携带的表**：传输时数据照搬，行为差异由用户自行确认（文档提示即可）。
- **SQL dump 的跨方言导出**（导出 MySQL 库为 PG 风格 .sql）：dump 始终用**源**方言渲染；要落到异构目标请走 DB→DB 传输。dump 文件里的 `'\xhex'` bytea 字面量依赖 PG `standard_conforming_strings=on`（8.2+ 默认），不再兼容更老配置。
